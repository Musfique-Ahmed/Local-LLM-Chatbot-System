# **1\. Product Overview**

This document defines the product requirements for a locally-hosted conversational AI chatbot system. The system uses a Python AsyncIO TCP server, Redis for per-user conversation history, and a locally-running large language model (Gemma 3 4B via Ollama) to deliver a private, low-latency chat experience with no dependency on external cloud APIs.

The system is designed as a phased build. Version 1 targets a single-user MVP under \~500 lines of Python; subsequent versions scale to hundreds of concurrent users, streaming output, and optional RAG capabilities.

## **1.1 Problem Statement**

Existing chatbot integrations rely on cloud-hosted LLM APIs (OpenAI, Anthropic, etc.), introducing latency, cost, privacy concerns, and rate limits. This project delivers a fully offline, self-hosted alternative suitable for university lab environments, portfolio demonstration, and privacy-sensitive use cases.

## **1.2 Goals**

* Build a working local chatbot with persistent conversation memory per user

* Keep the initial implementation under 500 lines of Python

* Demonstrate distributed systems concepts: networking, caching, and LLM integration

* Establish a clear upgrade path from single-user MVP to 1,000+ concurrent users

* Avoid cloud dependencies in the core architecture

## **1.3 Non-Goals**

* No web browser interface in V1 (TCP socket only)

* No fine-tuning or training of models

* No integration with external APIs or third-party LLM services

* No persistent user authentication in Phase 1

# **3\. System Architecture**

## **3.1 High-Level Architecture**

| User Layer | TCP Client — sends JSON messages, receives JSON responses |
| :---- | :---- |
| **Transport Layer** | Python AsyncIO TCP Server — accepts connections, routes messages |
| **Memory Layer** | Redis — stores per-user conversation history as JSON lists |
| **Inference Layer** | Ollama (local) running Gemma 3 4B — generates completions |
| **Serialization** | JSON — all inter-component communication |

## **3.2 Request Flow**

1. Client sends JSON payload: { user\_id, message }

2. TCP server receives and deserializes the message

3. Server fetches conversation history from Redis key chat:{user\_id}

4. Server builds a prompt from history \+ new message

5. Prompt is sent to Ollama's local HTTP API (localhost:11434/api/generate)

6. Gemma 3 4B generates a response (streaming or full)

7. Server appends both turns to Redis and trims to MAX\_MESSAGES

8. Response is serialized to JSON and sent back to the client

## **3.3 Directory Structure**

**chatbot/**

| server.py | AsyncIO TCP server — connection handling and orchestration |
| :---- | :---- |
| **llm.py** | Ollama HTTP client — prompt building and completion calls |
| **redis\_store.py** | Redis wrapper — get/set/trim conversation history |
| **config.py** | Environment constants — ports, model name, MAX\_MESSAGES |
| **clients/tcp\_client.py** | Reference CLI client for local testing |
| **logs/** | Rotating log files for debug and request tracing |

# **7\. Phased Rollout Plan**

## **Phase 1 — Single-User MVP**

Scope: One TCP connection, synchronous LLM call, Redis memory, JSON I/O. Target: under 500 lines of Python.

* server.py with asyncio.start\_server

* llm.py calling Ollama /api/generate (non-streaming)

* redis\_store.py with get/set/trim helpers

* cli tcp\_client.py for manual testing

## **Phase 2 — Multi-User Support**

Scope: 100 concurrent users via AsyncIO. Each connection isolated by user\_id Redis key.

* Ensure all coroutines are non-blocking

* Unique Redis key per user: chat:{user\_id}

* Connection pooling for Redis client

## **Phase 3 — Optional Auth**

Scope: Simple token-based session validation on connect. Adds user identity without a full auth service.

* Token passed in initial handshake message

* Server validates against a static config or lightweight lookup

## **Phase 4 — Streaming Responses**

Scope: Token-by-token streaming from Ollama to client. User sees output generated live.

* Enable streaming=true in Ollama API call

* Yield tokens over TCP connection as newline-delimited JSON

* Client buffers and prints progressively

## **Phase 5 — Context Window Management**

Scope: Enforce MAX\_MESSAGES cap on Redis history to prevent prompt overflow.

* redis\_store.trim(user\_id, MAX\_MESSAGES) after every save

* Default MAX\_MESSAGES \= 20 (configurable via config.py)

## **Future: RAG and Function Calling (V4)**

* Add vector database (e.g. ChromaDB) for document search

* Implement retrieval-augmented generation pipeline

* Add function calling for structured tool use

# **9\. Risks & Mitigations**

| Risk | Severity | Mitigation |
| :---- | :---- | :---- |
| Ollama/Gemma latency too high for real-time feel | **High** | Implement streaming (Phase 4); use GPU if available; tune max\_tokens |
| Redis history grows unbounded over long sessions | **Medium** | Trim to MAX\_MESSAGES after every write (Phase 5\) |
| Concurrent users block on synchronous LLM calls | **High** | Use asyncio and non-blocking HTTP calls with aiohttp in Phase 2 |
| Redis connection not available on startup | **Medium** | Add health-check retry loop on server start; Docker Compose depends\_on |
| Prompt injection via crafted user messages | Low (V1) | Sanitize user input before prompt assembly; add system message guardrails |
